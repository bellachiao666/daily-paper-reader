## 摘要
递归语言模型（RLM）表明，模型调用上的递归是长上下文推理的有效策略，而生产环境中的编码智能体已经开始编写能够大规模生成子智能体的代码，最近的一个例子是Anthropic的动态工作流。我们命名并研究这两条工作路线之间的模式，其中递归单元是一个完整的智能体框架，包含文件系统工具、代码执行和规划，而不是一个没有工具的模型调用。我们称之为递归智能体框架（RAH），并将其视为框架递归——RLM的模型递归的代码优先扩展。父智能体生成并运行一个可执行脚本，该脚本并行生成子智能体框架以处理细粒度工作负载，并使用结构化函数调用来处理小子任务。我们在长上下文推理上提供了受控评估。在将主干模型固定为GPT-5以匹配已发表的Codex和RLM基线时，RAH在Oolong-Synthetic数据集（199个样本，13个上下文长度区间，最长4M token）上将Codex编码智能体基线从71.75%提升至81.36%，这一提升归因于框架本身而非模型。使用更强的主干模型Claude Sonnet 4.5，同样的设计达到了89.77%。

## Abstract
Recursive language models (RLMs) showed that recursion over model calls is an effective strategy for long-context reasoning, and production coding agents have begun to write code that spawns subagents at scale, most recently in Anthropic's dynamic workflows. We name and study the pattern between these two lines of work, where the recursive unit is a full agent harness with filesystem tools, code execution, and planning rather than a model call with no tools. We call this the Recursive Agent Harness (RAH) and frame it as harness recursion, the code-first extension to the model recursion of RLMs. A parent agent generates and runs an executable script that spawns subagent harnesses in parallel for fine-grained workloads and uses structured function calls for small subtasks. We provide a controlled evaluation on long-context reasoning. With the backbone held fixed at GPT-5 to match the published Codex and RLM baselines, RAH improves the Codex coding-agent baseline from 71.75% to 81.36% on Oolong-Synthetic (199 samples, 13 context-length buckets up to 4M tokens), a gain attributable to the harness rather than the model. With a stronger backbone, Claude Sonnet 4.5, the same design reaches 89.77%.

## 1. 核心问题与整体含义（研究动机和背景）
- **研究动机**：长上下文推理任务（如聚合百万级token中的信息）面临双重挑战——既要大规模文档导航能力，又要对数千个独立条目进行逐条LLM推理。现有方法各执一端：
  - **编码智能体**（如Codex）拥有文件系统、代码执行等完整工具，但常依赖正则启发式处理入口，缺乏逐条LLM推理。
  - **递归语言模型（RLM）** 通过递归调用模型自身实现分解，但没有工具访问、文件系统或代码执行能力。
  - 生产环境中（如Anthropic动态工作流）已出现通过代码生成子智能体的实践，但缺乏系统研究和基准评估。
- **研究目标**：命名并评估一种新模式——**递归智能体框架（Recursive Agent Harness, RAH）**，其递归单元是完整的智能体框架（含工具、文件系统、规划），而非裸模型调用。目标是证明这种“框架递归”比模型递归或单智能体编码更有效。

## 2. 方法论：核心思想、关键技术细节
- **核心思想**：父智能体（parent agent）接收任务后，决定如何分解工作。对于细粒度工作负载，父智能体生成一个**可执行脚本**，在脚本中通过`Task()`对象并行生成多个子智能体框架（subagent harnesses），每个子智能体拥有独立的上下文窗口、文件系统工具、代码执行和规划能力。对于小规模子任务（1-5个条目），则直接使用结构化JSON函数调用（Tool call）生成子智能体。
- **关键技术细节**：
  - **代码执行路径**：父智能体通过shell工具执行生成的Python脚本（使用`asyncio.gather`并行运行），脚本中每个`Task()`调用实例化一个完整的子智能体。脚本路径绕过API的每轮工具调用限制，可扩展至数千个并行子智能体。
  - **JSON工具调用路径**：用于少量条目，直接通过结构化函数调用生成子智能体，避免脚本生成开销。父智能体根据条目数量自动选择路径。
  - **子智能体架构**：每个子智能体是完整框架，具备`read_file`, `write_file`, `ls`, `glob`, `grep`, `execute`以及网络搜索工具，并包含规划步骤。子智能体之间彼此隔离，无共享内存或通信通道。子智能体自身也可递归生成子子智能体（深度可配置，默认3层），实现多级分解。
  - **输出聚合**：所有子智能体完成后，父智能体读取共享输出文件中的结构化JSON记录进行聚合。
- **算法流程**（文字说明）：
  1. 父智能体接收任务，检查文档确定工作负载规模。
  2. 若条目数 > 5，选择代码执行路径：编写Python脚本，脚本中为每个条目创建`Task()`并收集为列表，通过`asyncio.gather`并行执行。若条目数 ≤ 5，直接调用JSON工具生成子智能体。
  3. 每个子智能体运行自己的规划-工具-反思循环，完成任务后写入结构化JSON输出。
  4. 父智能体等待所有子智能体完成（或通过聚合脚本读取输出文件），提取最终答案。

## 6. 主要结论与发现
- **核心结果**：在固定GPT-5骨干下，RAH将最强先前结果（Codex编码智能体）从71.75%提升至81.36%，提升9.61个百分点（95% CI [4.2, 14.8]），可归因于框架而非模型。使用更强骨干Claude Sonnet 4.5时达到89.77%。
- **按答案类型**：语义类型（USER, COMPARISON, LABEL）均超过86%；NUMERIC仅69.33%，受评分函数惩罚抑制；DATE因样本量小（5例）仅60%。
- **按上下文长度**：Sonnet 4.5在多数长度桶上超过86%，最高达100%（64K桶），直到4M仍保持76.7%。GPT-5在长上下文（262K、1M）有明显下降，受NUMERIC密集实例影响。
- **失败模式**：少数情况父智能体直接回答未生成脚本（集中在长上下文）；NUMERIC评分惩罚过度；DATE样本量小。
- **对比策略**：RAH在需要逐条推理和大规模并行的任务上优于RLM和单编码智能体。框架递归是模型递归的代码优先扩展。
