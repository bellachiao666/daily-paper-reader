## 摘要
基于LLM的智能体可以通过将推理与环境反馈相结合来解决多步交互任务，然而每一轮都从相同的固定上下文开始，且任何在过程中发现的有用策略在任务结束后都会丢失。现有方法要么将学习限制在当前任务上，要么将所有经验汇集到一个单一的未分类存储中，而不区分知识类型、通过使用来追踪质量，或平衡库中仍然缺乏的内容。我们引入了统一上下文进化（UCE），这是一个无梯度框架，将智能体经验外化为一个不断演化的类型化可进化上下文单元（ECU）库。UCE将经验分解为四种互补类型（记忆、策略、工作流和技能），每种类型从轨迹中根据特定类型条件生成，在决策时检索，通过反复使用结果评分，并在不再有价值时进行修剪。一个调度模块将每一轮的生成本分配给库中最薄弱的类型。在两个交互基准测试中，UCE将ALFWorld的成功率从75.4%提高到96.3%，将WebShop任务得分从45.1%提高到61.3%，并且累积的库可以无需重新训练地转移到替代的智能体主干上。

## Abstract
LLM-based agents can solve multi-step interactive tasks by combining reasoning with environment feedback, yet each episode starts from the same fixed context and any useful strategy discovered along the way is lost once the task ends. Existing approaches either limit learning to the current task or pool all experience into a single untyped store, without distinguishing knowledge types, tracking quality through use, or balancing what the library still lacks. We introduce Unified Context Evolution (UCE), a gradient-free framework that externalizes agent experience into an evolving library of typed Evolvable Context Units (ECUs). UCE decomposes experience into four complementary types (Memory, Strategy, Workflow, and Skill), each generated from trajectories under type-specific conditions, retrieved at decision time, scored through repeated usage outcomes, and pruned when no longer valuable. A scheduling module allocates each cycle's generation budget toward the types where the library is weakest. Across two interactive benchmarks, UCE raises ALFWorld success from 75.4% to 96.3% and WebShop task score from 45.1% to 61.3%, and the accumulated library transfers to alternative actor backbones without retraining.

### 1. 核心问题与整体含义（研究动机和背景）

- **核心问题**：当前基于大语言模型（LLM）的智能体（Agent）在执行多步交互任务时，每一轮任务都从相同的固定上下文（如系统指令、少样本示例）开始。智能体在任务过程中发现的有用知识（如环境机制、决策策略、操作流程）在任务结束后随即丢失，导致经验无法跨任务积累和复用，造成计算资源和能力的浪费。
- **研究背景**：现有解决方法存在不足：
    - **强化学习方法**（如WebRL, RAGEN）需要访问模型权重并承担高昂的训练成本。
    - **单任务内自适应方法**（如Reflexion, Self-Refine）仅能在当前任务中改进，任务结束后改进随之消失。
    - **跨任务知识库方法**（如ExpeL, Voyager, Agent Workflow Memory）或将所有知识存入单一无类型库（导致检索不精确），或只存储单一类型知识（如仅技能或仅工作流），且通常使用一次性LLM评估质量而非追踪实际使用效果，也无法动态平衡知识库中不同部分的成长。

### 2. 方法论：统一上下文进化（Unified Context Evolution, UCE）

核心思想是将智能体的经验外部化为一个**类型化、可进化的知识库**，让“上下文”本身成为学习的场所，无需更新模型参数。

- **核心组件：可进化上下文单元（Evolvable Context Units, ECUs）**
    - ECU是库中的基本知识单元，是一个五元组：`e = (ze, τe, we, ce, fe)`
        - `ze`: 功能类型（Memory, Strategy, Workflow, Skill）
        - `τe`: 适用的任务类型（Skill类型为`∅`表示跨类型适用）
        - `we`: 自然语言检索条件（用于匹配，不注入提示词）
        - `ce`: 注入到Agent提示中的知识内容
        - `fe`: 基于使用结果的适应度分数
    - **四种功能互补的ECU类型**:

| 类型（Type） | 定义问题 | 生成条件 | 代表知识 |
| :--- | :--- | :--- | :--- |
| **Memory** | 环境机制如何运作？ | 单条轨迹 | “用水槽盆可以清洁物品，无需打开水龙头。” |
| **Strategy** | 在特定条件下，Agent应选择什么？ | 同一任务类型的多条轨迹 | “如果放置失败，打开容器并重试。” |
| **Workflow** | Agent应遵循什么执行顺序？ | 同一任务类型的成功轨迹 | “找到目标，转换它，然后将其放在目标容器中。” |
| **Skill** | 如何执行一个可复用的操作？ | 多任务类型的轨迹 | “搜索容器，打开封闭的容器，找到目标后拿走。” |

- **关键技术细节**：
    - **检索**：执行任务前，根据任务描述和类型，对ECU进行打分并排序。打分函数结合语义相似度、类型匹配奖励和质量偏差。不同类型独立排序，确保每种类型在Agent提示中都有展现，防止某一类型主导。
    - **适应度分数（Fitness Score）**：`fe = (se + 1) / (ue + 2)`，其中`ue`是累积使用次数，`se`是累积结果信用（成功+1，失败+0，连续奖励则加奖励值）。新ECU的初始分数为0.5。还维护了针对特定任务类型的条件适应度分数。
    - **知识库生命周期管理**：通过 `Evaluate → Collect → Schedule → Generate → Cleanup` 五个阶段循环进行。
        1.  **Evaluate**: 使用当前库在留出评估集上运行，记录成功率（不更新适应度）。
        2.  **Collect**: 使用当前库在不相交的收集集上运行，记录轨迹并更新适应度分数。
        3.  **Schedule (KYS)**: 基于库的当前状态进行生成本预算分配。
        4.  **Generate**: 根据KYS分配的生成本，使用类型特定的提示（含质量反馈和重排测试）从收集的轨迹中提取新的ECU。
        5.  **Cleanup**: 移除使用次数足够但分数低、或重复的ECU。

- **知识产量调度（Knowledge Yield Scheduling, KYS）**:
    - KYS算法是预算分配的核心，通过三层分析决定生成本分配：
        - **可行性过滤**：判断每种ECU类型是否还有生成空间（如任务类型是否已被充分覆盖）。
        - **需求评估**：量化每种类型当前的知识缺口。对于记忆和策略，需求包括覆盖缺口和质量缺口；对于工作流，需求仅关注覆盖缺口；对于技能，需求取决于已有技能数量。
        - **产量追踪**：维护每种类型历史回报率的指数移动平均（EMA），评估新生成的ECU对性能提升的贡献。
    - 最终，通过一个动态调整的权重`α_c`融合需求评估和产量追踪，为每种可行类型分配生成本。

### 6. 主要结论与发现

1.  **显著性能提升**：UCE在ALFWorld上将成功率从75.4%提升至96.3%（峰值），在WebShop上将任务分数从45.1%提升至61.3%，购买成功率从30.0%提升至42.0%。

### 7. 优点

1.  **无梯度学习**：不更新任何模型权重，仅通过管理上下文实现智能体自我改进，适用于任何基于API的闭源LLM，应用范围广。
2.  **细粒度知识管理**：创新的四类型ECU设计（Memory, Strategy, Workflow, Skill），将经验结构化，实现了知识的精确定位、检索和互补，避免了混合库的信息杂乱。
3.  **动态、自适应的生命周期**：通过基于使用结果的适应度分数、知识产量调度（KYS）和定期清理，实现了知识库的“优胜劣汰”和自适应成长，自动将资源投入到最需要的领域。
4.  **良好的设计选择验证**：实验通过消融、案例分析和错误分析，有力地验证了其方法中的各项设计选择（如类型划分、KYS、修剪策略）的有效性。

### 8. 不足与局限

1.  **场景覆盖有限**：实验仅在两个离散动作空间的文本环境中进行，结果在连续动作、多模态观测或更复杂规划域（如AndroidWorld, VisualWebArena）的通用性尚未验证。
2.  **模型依赖性未解耦**：所有实验依赖单一模型对（`gpt-4.1-mini`作为Agent, `gpt-5.2`作为生成器），无法区分性能提升来自UCE框架本身、生成器的强大蒸馏能力，还是环境的规律性。需要独立的生成器模型变化实验来分离这些因素。
3.  **任务规模与长期演化**：任务池相对较小（134/100个任务），规划信号和多类型框架的调度信号均基于此估算。在更大规模任务集和更长的演化周期下，库是否能够持续改进、保持稳定或最终退化，仍是开放问题。
4.  **错误分析揭示的固有边界**：UCE主要捕捉“要了解什么”和“要采用什么规则/流程”，对于需要精确操作（如从属性面板选择正确尺码）的错误无能为力，这揭示了基于提示的知识与基于策略优化的强化学习方法之间的根本差异。
5.  **手工设计的类型**：四种ECU类型是手工设计的，缺乏自适应能力。一个能够自动发现并创建新类型知识的机制将是更有前瞻性的方向。
