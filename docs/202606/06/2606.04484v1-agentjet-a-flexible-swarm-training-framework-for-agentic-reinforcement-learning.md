## 摘要
我们提出AgentJet，一个用于大语言模型（LLM）智能体强化学习的分布式群体训练框架。与将智能体部署与模型优化紧密耦合的集中式框架不同，AgentJet采用解耦的多节点架构，其中群体服务器节点托管可训练模型并在GPU集群上运行优化，而群体客户端节点在任意设备上执行任意智能体。这种设计提供了集中式框架难以支持的能力：（1）异构多模型强化学习，使得能够训练以多个LLM为大脑的异构多智能体团队；（2）具有隔离智能体运行时的多任务鸡尾酒训练；（3）容错执行，防止外部环境故障中断训练过程；（4）实时代码迭代，允许通过替换群体客户端节点在训练期间编辑智能体。为了支持多模型、多轮次和多智能体场景下的高效强化学习，AgentJet引入了一个带有时间线合并的上下文追踪模块，该模块整合冗余上下文并实现了1.5-10倍的训练加速。最后，AgentJet引入了一个自动化研究系统，该系统以研究主题为输入，并在大规模集群上自主进行长周期、多天的强化学习研究。通过利用群体架构，该系统在执行过程中无需人工干预即可复现强化学习研究者的关键探索工作流程。

## Abstract
We present AgentJet, a distributed swarm training framework for large language model (LLM) agent reinforcement learning. Unlike centralized frameworks that tightly couple agent rollouts with model optimization, AgentJet adopts a decoupled multi-node architecture in which swarm server nodes host trainable models and run optimization on GPU clusters, whereas swarm client nodes execute arbitrary agents on arbitrary devices. This design provides capabilities that are difficult to support in centralized frameworks: (1) heterogeneous multi-model reinforcement learning, enabling the training of heterogeneous multi-agent teams with multiple LLM as brains; (2) multi-task cocktail training with isolated agent runtimes; (3) fault-tolerant execution that prevents external environment failures from interrupting the training process; and (4) live code iteration, which allows agents to be edited during training by replacing swarm client nodes. To support efficient RL in multi-model, multi-turn, and multi-agent settings, AgentJet introduces a context tracking module with timeline merging, which consolidates redundant context and achieves a 1.5-10x training speedup. Finally, AgentJet introduces an automated research system that takes a research topic as input and autonomously conducts long-horizon, multi-day RL studies on large-scale clusters. By leveraging the swarm architecture, this system reproduces key exploratory workflows of RL researchers without human intervention during execution.

## 论文的核心问题与整体含义

**研究动机与背景**  
现有的大语言模型（LLM）强化学习训练框架（如 OpenRLHF、veRL）主要针对单轮或短交互场景，将智能体部署与模型优化紧密耦合。当 LLM 向长时域、多工具、多智能体交互演进时，这种耦合带来了以下瓶颈：

*   **运行时脆弱**：智能体依赖的外部环境（如 Docker、浏览器、API）一旦故障，会中断整个训练管道并丢失未保存的进度。
*   **调试摩擦**：修改智能体代码或奖励函数需要重启整个训练过程（含模型加载），导致 5–10 分钟的迭代周期。
*   **多模型限制**：当前系统主要优化单一策略模型，而真实场景中常需异构多智能体系统（如 7B 执行、32B 规划），但现有框架难以支持非共享参数的多模型训练。
*   **上下文冗余**：多轮交互中对话上下文大量重复（系统提示、工具定义等），在策略梯度更新时浪费计算资源。
*   **环境锁死**：多任务鸡尾酒训练要求任务间有强隔离，但单体架构难以满足。

**整体含义**  
AgentJet 旨在通过彻底解耦智能体运行时与训练基础设施，提供一个灵活、可扩展、容错的分布式群体训练框架，支持异构多模型、多任务、在线代码迭代和自动化研究，推动 LLM 智能体的强化学习落地。

## 方法论：核心思想、关键技术细节

**核心思想**  
采用 **集群架构（Swarm Architecture）**，将训练平面与智能体执行平面分离为两类节点：

*   **Swarm Server（群服务器）**：托管模型权重、执行策略梯度更新（基于 veRL）、提供 OpenAI 兼容推理 API（vLLM/SGLang），并管理上下文追踪。
*   **Swarm Client（群客户端）**：轻量、无 GPU 要求，可在任意设备运行，执行任意智能体工作流、计算奖励、读写数据集，并通过 API 与服务器交互。

**关键技术细节**

1.  **异构多模型强化学习**：多个 Swarm Server 可分别训练不同模型（如 Qwen3-32B 和 Qwen3-14B），客户端按角色路由，实现非共享参数的多智能体训练。
2.  **多任务鸡尾酒训练**：不同任务（如 AppWorld、AIME）运行在独立客户端容器/环境中，统一向 Swarm Server 发送轨迹，服务器联合优化策略。
3.  **上下文追踪与时间线合并（Timeline Merging）**：
    *   每个 episode 内记录所有 LLM 调用的消息块、token ID、对数概率和损失掩码。
    *   **时间线合并算法**：从后向前依次尝试将较短时间线合并到较长时间线中，消除重叠前缀。支持 token 级别和 text 级别匹配策略（后者更激进，容忍 tokenizer 偏移）。实验显示可带来 **1.5–10 倍训练加速**，且不影响训练质量。
4.  **灵活的样本批处理策略**：
    *   提供五种触发策略（C1–C5），包括：任务完成数、有效任务数、所有客户端同意、任意客户端同意等，支持动态采样（如 DAPO 的零优势丢弃）。
5.  **框架无关集成**：通过 OpenAI 兼容端点，支持任意智能体框架（LangChain、AgentScope、MetaGPT、原始 HTTP 等），无需修改智能体代码。
6.  **自动化研究（A3R）**：构建两层智能体（Leader + Worker），Leader 制定多阶段实验计划，Worker 在 GPU 集群上并行执行。通过实验蓝图（Blueprint）描述目的、代码、配置、命令等，支持故障恢复和结果汇总。

## 主要结论与发现

1.  **Swarm 架构有效**：实现解耦训练，支持异构多模型、多任务、容错和在线代码迭代。
2.  **时间线合并显著加速**：AppWorld 上实现 6.25 倍加速，且不损害训练质量。
3.  **非共享参数训练优于共享参数**（狼人杀）：三个狼人各自用独立 LoRA 模型达到 66.5% 胜率，高于共享参数的 64.7%，原因是行为多样化使村民更难识别。
4.  **鸡尾酒训练可行但存在任务不对称**：联合训练下的 AIME 奖励几乎等同于单独训练，但 AppWorld 有约 10 点的差距（文中解释为工具使用轨迹更易被稀释），但鸡尾酒训练节省了总成本且产生通用模型。
5.  **框架无关性得到验证**：四种不同智能体框架的最终评估分数差异仅 0.025，证明捕获机制稳定。
6.  **自动化研究（A3R）成功完成复杂案例**：自主进行 19 个实验的批大小研究，发现最佳策略为 bs=4, max_response_length=12000（较初始配置提升 pass@1 8.3 点），并得出“响应长度比批大小更关键”的结论。

## 优点

*   **架构创新**：彻底解耦智能体执行与模型优化，为训练异构多智能体系统和多任务提供天然支持。

## 不足与局限

*   **应用限制**：需要部署多个 Swarm Server 和 Client，增加了运维复杂性，且客户端可能引入额外的网络延迟。
