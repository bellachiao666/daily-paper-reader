## 摘要
自演化的基于LLM的智能体主要通过改变其智能体控制框架（agent harness）来改进，该框架是围绕基础模型的结构化执行层，包括提示、记忆、工具、中间件、运行时状态以及模型-工具交互循环。现有评估通常将此过程简化为孤立的任务分数或单一的顺序曲线，掩盖了更新是否产生可复用的改进、是否过拟合近期任务、是否增加成本或损害旧有行为。我们提出SEAGym，一个评估环境，用于衡量智能体控制框架在训练、验证、测试、重放和成本记录中的更新。SEAGym将兼容Harbor的基准测试转化为动态自演化任务源，包含训练批次、冻结的更新验证、保留的同分布和跨分布迁移视图、重放诊断以及保存的快照和指标记录。在Terminal-Bench 2.0和HLE上实例化SEAGym，我们在共享的周期/批次协议下比较ACE、TF-GRPO和AHE。结果表明，这些评估视图提供了关于演化过程的互补信号：频繁更新可能无法提升保留性能，有用的中间快照可能随后崩溃，而源多样性和模型后端可能影响控制框架的可靠性。

## Abstract
Self-evolving LLM-based agents improve mainly by changing their agent harness: the structured execution layer around a base model, including prompts, memory, tools, middleware, runtime state, and the model-tool interaction loop. Existing evaluations often reduce this process to isolated task scores or a single sequential curve, obscuring whether an update produces reusable improvement, overfits recent tasks, increases cost, or harms older behavior. We introduce SEAGym, an evaluation environment for measuring agent harness updates across training, validation, test, replay, and cost records. SEAGym turns Harbor-compatible benchmarks into dynamic self-evolution task sources with train batches, frozen update-validation, held-out ID and OOD transfer views, replay diagnostics, and saved snapshot and metric records. Instantiating SEAGym on Terminal-Bench 2.0 and HLE, we compare ACE, TF-GRPO, and AHE under a shared epoch/batch protocol. The results show that these evaluation views provide complementary signals about the evolution process: frequent updates may fail to improve held-out performance, useful intermediate snapshots may collapse later, and source diversity and model backend can affect harness reliability.

## 2. 论文提出的方法论：核心思想、关键技术细节

### 核心思想
- 将自演化过程建模为**类MDP评估过程**：状态 \( s_t \) 包含当前智能体快照（基础模型 + 可变框架状态）、调度位置和任务上下文。环境采样任务批次，智能体执行任务产生轨迹并接收反馈，然后应用其自身的更新规则 \( H_{t+1} = U(H_t, B_t, T_t, F_t) \)。SEAGym定义观测环境（任务分布、反馈可见性、调度、评估视图），而策略和更新规则留给各方法。
- 关键区分：智能体快照 \( A_t = (M, H_t) \)，其中 \( M \) 是固定基础模型和不可变组件，\( H_t \) 是可变的框架状态（提示、记忆、技能、工具、中间件等）。自演化只改变 \( H_t \)，不改变基础模型权重。

### 关键技术细节
- **演化调度参数化**：支持在线、单任务、批次和周期式自演化设置，通过调度参数（状态持久性、任务复用、批次大小、更新次数、评估时机）统一表示。这使得不同方法可以在同一协议下公平比较。
- **数据划分与评估视图**：
  - 基础划分：保留传统的训练/验证/测试划分，控制任务可见性。
  - 评估视图从这些划分中实例化，回答不同问题：
    - **更新验证（Update-validation）**：追踪中间快照在验证集上的表现（不用于修改更新）。
    - **同分布迁移（ID transfer）**：在保留的源分布测试任务上评估最终快照。
    - **跨分布迁移（OOD transfer）**：在目标域测试集上评估泛化能力。
    - **重放/保留（Replay/Retention）**：检查更新后的框架是否保持或恢复了旧行为（遗忘/回归）。
- **基准与基线集成**：
  - 复用Harbor框架作为任务执行后端（容器化环境、验证器、并行执行）。
  - 方法侧：分离**rollout**（在当前框架状态下执行任务并返回轨迹批次）和**update**（使用轨迹批次应用方法的原生更新规则并保存新框架状态）。这允许不同的自演化方法（如ACE、TF-GRPO、AHE）通过薄封装连接到SEAGym，保留各自原生更新语义。
- **指标与保存记录**：主要指标包括成功率（Perf, SR）、更新验证增益（UVG）、同分布增益（IDG）、跨分布增益（OODG）、遗忘率（FR）以及token消耗、工具调用次数、时间成本等。保存轨迹引用、反馈、更新摘要、框架差异、快照和指标记录，支持离线过程诊断。

### 公式/算法流程（文字说明）
- 算法1（附录A.1）描述了SEAGym的epoch/batch评估流程：
  1. 从训练集中采样批次，选择验证/重放/最终视图。

  3. 对每个epoch，对每个训练批次：
     - 执行任务，收集轨迹和反馈。

## 6. 论文的主要结论与发现

- **更新机制影响显著**：
  - AHE（框架编辑）能同时提升验证、ID和OOD表现（+17.1% UVG, +9.1% IDG, +6.3% OODG），但其更大的编辑范围也带来了可靠性风险（中间快照可能崩溃）。
  - ACE（技能记忆）产生适中的ID和OOD增益（+2.9% UVG, +3.6% IDG, +2.5% OODG），但难以解决运行时行为故障。
  - TF-GRPO（分组经验）获得较大的源验证增益（+17.1% UVG），但OOD下降（-2.5% OODG）且rollout成本最高，表明适应不一定稳定泛化。
- **训练修复与遗忘分析**：AHE最终快照净修复9个初始失败任务，但失去4个初始成功任务；中间快照E16因运行时消息合约破坏导致严重崩溃（6/80成功，66个错误），后经修复恢复。这说明中间快照诊断的重要性。
- **批次大小非单调**：批次20是最佳平衡（+17.1%验证增益、+9.1% ID增益）；批次10和80均退步，原因分别是更新频率过高导致运行配置漂移，以及证据过多导致分析稀释。
- **源多样性**：混合源（Terminal-Bench+HLE）比仅HLE源更鲁棒；HLE-only训练最终快照崩溃（0%成功率），但中间快照有用，说明单一领域证据可能导致框架适应该领域特定失败模式而忽略其他。
- **跨模型转移**：同一后端演化后ID增益为正（+3.6~+9.1个百分点），但跨后端转移不稳定且不对称（例如DeepSeek演化的框架提升GLM ID但降低GPT-5.4 ID）。OOD增益普遍中性或为负，表明演化框架可能编码了训练后端的特定失败模式。

## 7. 优点

- **方法论亮点**：
  - 提出统一的评估环境，将自演化问题形式化，允许不同机制（在线、批次、周期）在同一协议下比较。
  - 分离了“任务执行”与“框架更新”，通过rollout/update接口实现模块化集成，减少了对基准测试和基线方法的侵入性修改。
  - 设计多维度评估视图：更新验证、ID迁移、OOD迁移、重放诊断，提供了比单一最终分数更丰富的演化过程信号。
  - 保存了快照、轨迹、更新摘要等工件，支持离线过程诊断和结果复现。
- **实验设计亮点**：
  - 深入分析了训练修复与遗忘行为（重放网格、快照级任务轨迹），揭示了中间崩溃和恢复的机制。
  - 消融实验系统探究了批次大小、源多样性和跨模型迁移的影响，提供了机制性解释（如更新预算、证据密度、运行时合约风险）。
  - 使用领域级别宏平均，避免大任务组主导结论。

## 8. 不足与局限

- **实验覆盖有限**：仅使用了两个基准（Terminal-Bench 2.0和HLE），未涵盖Web交互、桌面操作、多Agent协作等更广泛的agent领域。作者承认这一点，并指出未来可通过更多Harbor兼容任务源扩展。
- **未涉及模型权重更新**：论文聚焦于模型外部框架演化，未研究模型权重更新（如在线微调、强化学习）或混合系统。同一SEAGym环境可扩展支持此类设置，但当前实验未包括。
- **成本/覆盖权衡**：多视图设计需要保存和重新评估多个快照，增加了评估成本。未来可研究更高效的快照选择策略。
- **跨模型和OOD实验**：仅使用了三个模型后端和两个域，结论的泛化性有待更多后端和域组合验证。结果也表明，框架更新可能偏向训练后端的行为模式。
- **实验公平性细节**：虽然统一了协议，但不同方法的默认配置（如ACE的上下文长度、TF-GRPO的组大小）可能影响结果。论文未完全对齐所有超参数或计算预算（如ACE未报告更新token成本）。此外，部分运行时数据不完整（如TF-GRPO主运行时间未记录）。
- **缺乏统计学检验**：实验似乎仅运行了一次（或有限次数），未报告置信区间或多次运行的标准差。特别是批次消融和跨模型结果可能存在随机性。
- **应用限制**：SEAGym作为评估框架不应暴露隐藏的验证器细节；若未来增加真实用户数据或专有存储库，需遵循许可和隐私要求。
