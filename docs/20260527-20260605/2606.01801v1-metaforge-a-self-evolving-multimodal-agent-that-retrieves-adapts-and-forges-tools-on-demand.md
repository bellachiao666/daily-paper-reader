## 摘要
多模态智能体通过工具使用在复杂推理任务上取得了显著进展，但仍受限于两个问题：静态预定义的工具库无法泛化到未见场景，而 indiscriminate 工具调用会带来冗余成本和噪声导致的错误。我们提出 MetaForge，一个多模态智能体框架，它学习何时调用工具以及如何按需进化其工具集。MetaForge 将智能体行为分解为四个耦合阶段：判断（决定是否需要使用工具）、检索（选择合适的工具）、适配（将工具参数与任务上下文对齐）和锻造（在线合成新技能并将其回收至工具库以供复用），形成一个闭环的 判断-检索-适配-锻造-回收 循环。一个统一的编排策略使智能体能够在直接回答、复用现有工具或锻造新工具之间进行选择。我们通过强化学习联合优化调用必要性、检索准确性、执行有效性和锻造技能的可复用性，并显式加入调用成本惩罚以阻止冗余调用。在12个基准测试中，MetaForge 在准确性、效率和泛化能力上持续超越16个基线，验证了从静态工具库到按需自我进化的范式转变。

## Abstract
Multimodal agents have achieved notable progress on complex reasoning tasks through tool use, yet remain limited by two issues: statically predefined tool inventories fail to generalize to unseen scenarios, and indiscriminate tool invocation incurs redundant cost and noise-induced errors. We propose MetaForge, a multimodal agent framework that learns when to invoke tools and how to evolve its toolset on demand. MetaForge factorizes agentic behavior into four coupled stages: Decide (judging whether tool use is warranted), Retrieve (selecting suitable tools), Adapt (grounding tool parameters in task context), and Forge (synthesizing new skills online and recycling them into the tool library for reuse), forming a closed judge-retrieve-adapt-forge-recycle loop. A unified orchestration policy enables the agent to choose among answering directly, reusing existing tools, or forging new ones. We jointly optimize invocation necessity, retrieval accuracy, execution effectiveness, and forged-skill reusability via reinforcement learning, with an explicit invocation-cost penalty discouraging redundant calls. Across 12 benchmarks, MetaForge consistently surpasses 16 baselines in accuracy, efficiency, and generalization, validating a paradigm shift from static tool inventories to on-demand self-evolution.

## 1. 核心问题与整体含义（研究动机和背景）

当前多模态智能体通过外部工具（如OCR、图像处理模块）增强复杂推理能力，但存在两大局限：

- **静态预定义工具库无法泛化**：预先固定的工具集不能覆盖未见过的新场景或任务需求。
- **不加区分的盲目调用**：模型在不需要工具时也发起调用，造成冗余计算成本和噪声导致的错误。

上述问题导致四种典型失败模式：多余调用、工具-任务不匹配、参数错误、缺乏必要工具。论文旨在实现一个能够**按需自我进化**的智能体框架，使其能灵活决定何时调用工具、如何适配调用、以及在能力缺口出现时锻造新工具，从而完成从静态工具库存向按需动态演化的范式转变。

## 2. 方法论

### 2.1 核心思想
MetaForge将智能体行为建模为一个闭环的 **判断‑检索‑适配‑锻造**（Decide‑Retrieve‑Adapt‑Forge）流程，并结合强化学习统一优化决策策略。系统维护一个动态工具池，包含三类工具：可执行工具（如OCR、代码执行）、模型工具（如专用视觉模型）和锻造工具（按需生成的技能）。

### 2.2 关键技术细节

- **问题形式化**：输入为视觉+文本 \(x=[I,T]\)，策略 \(\pi_\theta\) 基于历史 \(h_t\) 和当前工具池 \(\mathcal{T}_t\) 选择动作 \(a_t\)，动作可以是调用工具、锻造工具或输出答案。工具统一接口为 \(\langle name, desc, schema, type \rangle\)。

- **在线工具锻造**：当现有工具均不满足要求时，策略触发 `create_skill` 动作，通过两阶段生成锻造工具：
  1. **规划阶段**：锻造模型 \(G_\phi\) 生成结构化计划（名称、描述、参数接口、脚本列表）。
  2. **实现阶段**：为每个脚本生成代码。锻造工具须通过编译执行验证 (\(v_{\text{exec}}\)) 和语义对齐验证 (\(v_{\text{sem}}\)) 后才可注册回工具池。

- **复合奖励与多轮GRPO优化**：对每个输入采样 \(M=8\) 条轨迹，设计四部分奖励：
  - **Retrieve/Adapt 奖励**：最终答案是否正确（1/0）。
  - **Decide 奖励**：仅在正确答案轨迹中，按调用延迟归一化评分，鼓励简洁调用路径。
  - **Forge 奖励**：鼓励成功锻造并复用已注册锻造工具（仅对正确答案轨迹生效）。
  - **Format 奖励**：检查输出格式是否合规（±0.2）。
  
  使用GRPO更新策略，通过组内标准化计算优势，并用裁剪目标函数防止过大更新。

- **容量约束工具池管理**：设置最大容量 \(K=50\)，通过语义去重（合并功能重叠的锻造工具）和基于使用频率的保留策略（优先保留高频使用的现有工具及当前需求大的新锻造工具）控制工具池规模。

## 6. 主要结论与发现

- MetaForge在12个基准上**准确率、效率和泛化性均一致优于所有16个基线**，在IID和OOD两种工具设置下均取得最佳或次佳结果。
- **在线工具锻造有效弥补能力缺口**：在IID工具下，当遇到OOD任务（如WebSight）时主动锻造任务特定工具；在OOD工具下则几乎不需要锻造，表明锻造行为与需求缺口精确对齐。
- **复合奖励各项均有贡献**：Format奖励最重要，缺失时准确率大幅下降；Forge奖励对OOD数据集关键；Decide奖励提升效率并减少中间错误。
- 方法**不依赖特定子模型架构**：在QwenVL和InternVL系列上均获得显著提升，且弱模型受益更大。
- **工具调用质量显著优于基线**：幻觉类错误极少（IID下仅14次，OOD下9次），失败主要是执行环境因素而非模型对工具的理解缺陷。

## 7. 优点

- **范式创新**：提出从静态工具库到“按需自我进化”的新范式，智能体能够在线识别能力缺口、即时锻造新工具并回收复用，突破了传统固定工具集的上限。
- **统一优化框架**：通过多轮GRPO和复合奖励机制，联合优化了工具使用的多个维度（正确性、效率、锻造行为、格式），无需人工设计规则。
- **工具池容量管理**：设计去重和基于使用频率的保留策略，有效控制工具池增长，避免上下文过载和注意力稀释。
- **鲁棒性验证充分**：不仅评估标准IID设置，还进行了OOD工具和OOD数据集双重泛化考验，并跨多个骨干模型验证方法通用性，实验设计严谨周密。
- **案例分析丰富**：提供了完整的多轮交互轨迹，直观展示了自适应调用、高效决策和按需锻造的具体行为模式，增强了方法的可解释性。

## 8. 不足与局限

- **锻造技能形式受限**：当前仅支持可执行脚本（Markdown规范+JSON schema+Python代码），适用于程序化任务（代码处理、图表分析、数值计算），无法处理需额外模型或外部资源的任务（如视频合成）。未来需探索更强锻造模型、更丰富技能形式和更可靠的验证机制。
- **评估任务范围有限**：12个基准均限于图像-文本模态，未涉及音频、视频、时间推理或交互式环境。在线锻造在更丰富的多模态设置中是否同样有效尚待验证。
- **效率优化单一**：当前效率目标仅关注延迟，未将工具调用的货币成本、上下文使用量等因素纳入优化目标。未来需在保持准确性的前提下，平衡成本、上下文占用和响应时间。
- **工具池容量上限为固定值**：实验设定K=50，但在持续进化的场景下，如何动态调整容量或进行更智能的淘汰策略未深入探讨。
