## 摘要
LLM智能体越来越多地通过编写代码来行动，但驱动智能体的运行时与模型编写的代码之间仍然存在分裂。运行时拥有循环、上下文和控制流，而模型对这些几乎没有发言权。让模型编写的代码塑造运行时本身将使智能体更具表现力，但也会加剧安全问题。模型可能因提示注入而偏离，调用错误的工具，或在中途失败并留下不一致的状态，当代码塑造运行时比仅表达单个动作时，每个此类故障的影响范围更广。我们提出LACUNA，一种用于智能体的编程模型，它在保持安全的同时消除了这种分裂。每个智能体动作是一个类型化调用 $\texttt{agent[T](task)}$，当执行到达时，LLM用代码填充它，并且代码在运行前会针对周围程序进行类型检查。由于每个动作作为一个整体被接受或拒绝，被拒绝的动作不会影响环境，其编译器诊断信息会驱动重试。相同的检查还限制了动作可能使用的工具和数据以及它们的流动方式。我们的原语将ReAct循环、子智能体、技能、并行分解和多模型规划表达为普通的控制流。我们在测试用例集、BrowseComp-Plus和$τ^2$-bench上评估LACUNA。在BrowseComp-Plus上，$8.6\%$的生成在运行前被拒绝，每个查询平均重试0.7次，智能体达到$27.1\%$的准确率。在$τ^2$-bench上，LACUNA在四个领域中使用一个能力强的模型解决了392个任务中的$76.0\%$，与基线智能体相当。

## Abstract
LLM agents increasingly act by writing code, yet a split persists between the runtime that drives the agent and the code the model writes. The runtime owns the loop, context, and control flow, and the model has little say over any of them. Letting model-written code shape the runtime itself would make agents more expressive, but it would also sharpen safety problems. A model can be diverted by a prompt injection, call the wrong tool, or fail partway and leave an inconsistent state, and each such failure reaches further when the code shapes the runtime than when it expresses a single action. We present LACUNA, a programming model for agents that closes this split while preserving safety. Each agent action is a typed call $\texttt{agent[T](task)}$ that the LLM fills with code when execution reaches it, and the code is type-checked against the surrounding program before it runs. Because each action is accepted or rejected as a whole, a rejected one leaves the environment untouched, and its compiler diagnostics drive a retry. The same check also bounds which tools and data an action may use and how they flow. Our primitive expresses ReAct loops, sub-agents, skills, parallel decomposition, and multi-model planning as ordinary control flow. We evaluate LACUNA on a collection of test cases, BrowseComp-Plus, and $τ^2$-bench. On BrowseComp-Plus, $8.6\%$ of generations are rejected before execution, with 0.7 retries per query on average, and the agent reaches $27.1\%$ accuracy. On $τ^2$-bench, LACUNA solves $76.0\%$ of $392$ tasks across four domains with a capable model, on par with the baseline agent.

## 1. 核心问题与整体含义

- **研究动机**：现有 LLM 智能体（如 ReAct）中，运行时拥有循环、上下文和控制流，而模型仅生成工具调用或代码片段。这种分裂限制了表达力；若让模型代码直接塑造运行时，可提升灵活性，但会引入严重安全风险：提示注入、错误工具调用、部分执行导致状态不一致。  
- **整体含义**：LACUNA 提出一种编程模型，将每个智能体动作视为一个“类型化程序洞”（typed hole），由 LLM 在运行时填充代码，并在执行前利用宿主语言编译器进行完整静态检查（包括类型匹配、作用域、能力使用）。被拒绝的动作不会执行任何部分，从而保证环境不变；编译器诊断反馈给 LLM 驱动重试。目标是在不牺牲安全的前提下融合运行时与模型代码。

## 2. 方法论

- **核心思想**：定义原语 `def agent[T](task: String): T`，其中 `T` 是期望结果类型，`task` 是自然语言描述。当执行到达该调用时，LLM 生成 Scala 代码，LACUNA 在相同词法上下文（包括变量、函数、工具、能力）中编译该代码，检查是否类型正确且遵守能力约束。若通过则执行，否则抛出编译异常或返回失败结果，驱动重试。  
- **关键技术细节**：  
  - 基于 Scala 3 的 `eval` 原语：将代码字符串在运行时编译如同手写代码，利用编译器类型检查和捕获检查（capture checking）。  
  - `eval` 自动捕获调用点处的绑定（变量、类型、封闭源代码），并在重新编译时使用相同编译选项。  
  - 原子性：整个片段一起编译，拒绝则任何语句都不执行。  
  - 能力安全：通过捕获检查限制生成代码能访问的资源（文件、网络等），并防止能力泄露到作用域外。  
  - 嵌套递归：`agent` 调用可以递归嵌套，每个嵌套有自身类型和上下文，形成子智能体、循环（ReAct）、并行等模式。  
- **无单独安全模块**：安全性完全由宿主编译器保证。

- 递归嵌套、并行、子智能体等模式可通过普通控制流实现，无需额外原语。

## 7. 优点

- **创新性融合**：将智能体动作建模为程序洞，利用宿主编译器静态保证安全，而非外部沙箱或启发式过滤。  
- **原子拒绝**：整个片段编译失败不执行任何语句，杜绝部分执行泄露。  
- **能力追踪**：通过捕获检查实现细粒度的权限和明文数据流控制，对抗注入。  
- **表达力**：递归嵌套支持 ReAct、子智能体、并行、多模型规划等，保持语言一致性。  
- **实现简洁**：仅需一个 `eval` 原语，wrapper `agentSafe` 实现重试逻辑，安全属性继承自编译器。  
- **实验全面**：覆盖类型检查、复杂工具、多轮对话、对抗攻击，对比多个模型和基线。

## 8. 不足与局限

- **类型正确 ≠ 语义正确**：编译通过的动作仍可能做错误算法或调用错误工具，无法保证正确性。  
- **权限取决于授予的作用域**：若开发者过度授权（如让 agent 访问网络能力但任务不需要），注入攻击依然可能滥用。能力安全不自动推断最小权限。  
- **依赖模型编码能力**：弱模型写不好 Scala 代码时重试成本高甚至无法完成任务（如 gemini-3.1-flash-lite 在电信领域 89% 生成被拒绝）。  
- **延迟与成本**：每次 agent 调用需一次模型推理 + 一次编译，且递归和重试使成本线性增加，不适合极低延迟场景。  
- **宿主语言限制**：依赖静态类型、捕获检查、运行时编译，不适合 Python 等动态语言生态（除非放弃安全保证）。  
- **安全模式必要**：需关闭反射和进程执行（safe mode），否则仍有逃逸。  
- **终止与资源消耗无法类型保证**：依赖运行时上限（嵌套深度、时间、重试次数），设置过松可能导致无限递归，过紧可能错误截断合法长任务。  
- **实验单次运行**：τ²-bench 未报告多次运行统计，可能降低结果可靠性。
