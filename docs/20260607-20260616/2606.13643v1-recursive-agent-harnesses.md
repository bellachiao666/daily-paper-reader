## 摘要
递归语言模型（RLM）表明，模型调用上的递归是一种有效的长上下文推理策略，而生产级编码代理已开始编写大规模生成子代理的代码，最近在Anthropic的动态工作流中也是如此。我们命名并研究了这两条工作线之间的模式，其中递归单元是一个带有文件系统工具、代码执行和规划能力的完整代理框架，而不是一个没有工具模型调用。我们将其称为递归代理框架（RAH），并将其定义为框架递归，即RLM模型递归的代码优先扩展。父代理生成并运行一个可执行脚本，该脚本并行生成子代理框架以处理细粒度工作负载，并使用结构化函数调用来处理小型子任务。我们在长上下文推理上提供了受控评估。在固定主干为GPT-5以匹配已发布的Codex和RLM基线的情况下，RAH在Oolong-Synthetic（199个样本，13个上下文长度桶，最长4M令牌）上将Codex编码代理基线从71.75%提升至81.36%，这一提升归因于框架而非模型。使用更强的主干Claude Sonnet 4.5，相同设计达到了89.77%。

## Abstract
Recursive language models (RLMs) showed that recursion over model calls is an effective strategy for long-context reasoning, and production coding agents have begun to write code that spawns subagents at scale, most recently in Anthropic's dynamic workflows. We name and study the pattern between these two lines of work, where the recursive unit is a full agent harness with filesystem tools, code execution, and planning rather than a model call with no tools. We call this the Recursive Agent Harness (RAH) and frame it as harness recursion, the code-first extension to the model recursion of RLMs. A parent agent generates and runs an executable script that spawns subagent harnesses in parallel for fine-grained workloads and uses structured function calls for small subtasks. We provide a controlled evaluation on long-context reasoning. With the backbone held fixed at GPT-5 to match the published Codex and RLM baselines, RAH improves the Codex coding-agent baseline from 71.75% to 81.36% on Oolong-Synthetic (199 samples, 13 context-length buckets up to 4M tokens), a gain attributable to the harness rather than the model. With a stronger backbone, Claude Sonnet 4.5, the same design reaches 89.77%.

## 1. 核心问题与整体含义（研究动机和背景）

- **研究动机**：长上下文推理任务（如从数百万令牌中提取信息）面临两难：编码代理（coding agents）有文件系统和代码执行能力，但通常将每个条目的推理简化为正则启发式，缺乏逐条LLM推理；递归语言模型（RLM）通过模型调用递归处理子切片，但没有工具访问（文件、代码执行等）。两者都有盲点，而生产系统（如Anthropic动态工作流）已经开始用代码生成子代理。因此需要系统命名并评估这种以完整代理框架为递归单元的模式。
- **整体含义**：提出“递归代理框架”（Recursive Agent Harness, RAH），将递归单元从模型调用升级为完整的代理框架（含工具、文件系统、规划）。这是RLM模型递归的代码优先扩展，填补了编码代理与RLM之间的空白，为可扩展的逐条理由提供通用范式。

## 2. 方法论：核心思想、关键技术细节

- **核心思想**：父代理（parent agent）通过两种方式生成子代理：
  - **代码执行路径**（code-execution spawning）：父代理编写并运行一个可执行脚本（如Python），脚本中创建多个`Task()`对象，通过`asyncio.gather`并行调用子代理框架。每个子代理是完整的代理框架（文件读写、ls、grep、代码执行、web搜索等），拥有独立上下文和工具。脚本路径绕过API的每轮调用限制，可扩展至数千个子代理。
  - **JSON工具调用路径**（JSON tool-call spawning）：对于1–5个条目，父代理直接通过结构化函数调用生成子代理，无需生成中间脚本。
- **递归性**：每个子代理本身具有和父代理相同的工具和生成能力，因此可递归生成子代理（深度限制默认3层），实现多级分解。
- **隔离与聚合**：子代理之间无共享内存或通信，各自写入指定输出文件，父代理读取汇总。
- **设计哲学**：将代码视为一等行动（code-as-action），而非固定工具模式。父代理根据工作负载自动选择生成路径。
