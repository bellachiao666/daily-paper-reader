## 摘要
自主智能体系统在部署后基本处于静态：它们无法从用户交互中学习，重复性的故障会持续存在，直到下一次人工驱动的更新提供修复。为此，自我进化的智能体应运而生，但所有进化都局限于文本可变的工件——技能文件、提示配置、记忆模式、工作流图——而智能体的框架本身保持不变。由于路由、钩子排序、状态不变性和调度存在于代码而非任何文本工件中，因此整个结构故障类别从文本层在物理上是不可达的。我们认为源码级适应是一种本质上更通用的媒介：它是图灵完备的，是每个文本可变范围的严格超集，能够确定性地生效而非依赖于基模型的顺从性，并且不会在长上下文漂移下退化。我们提出了MOSS，一个在生产级智能体基底上执行源码级自我重写的系统。每次进化都锚定在一个自动策划的生产故障证据批次上，并通过确定性的多阶段流水线进行；代码修改委托给一个可插拔的外部编码智能体CLI，而MOSS保留阶段排序和裁决。候选方案通过在临时试运行工作器中针对候选映像重放该批次进行验证，然后通过用户同意门控的就地容器替换以及健康探测门控的回滚进行升级。在OpenClaw上，MOSS在单次循环中无需人工干预即将四个任务的平均评分从0.25提升至0.61。

## Abstract
Autonomous agentic systems are largely static after deployment: they do not learn from user interactions, and recurring failures persist until the next human-driven update ships a fix. Self-evolving agents have emerged in response, but all confine evolution to text-mutable artifacts -- skill files, prompt configurations, memory schemas, workflow graphs -- and leave the agent harness untouched. Since routing, hook ordering, state invariants, and dispatch live in code rather than in any text artifact, an entire class of structural failure is physically unreachable from the text layer. We argue that source-level adaptation is a fundamentally more general medium: it is Turing-complete, a strict superset of every text-mutable scope, takes effect deterministically rather than through base-model compliance, and does not erode under long-context drift. We present MOSS, a system that performs self-rewriting at the source level on production agentic substrates. Each evolution is anchored to an automatically curated batch of production-failure evidence and proceeds through a deterministic multi-stage pipeline; code modification is delegated to a pluggable external coding-agent CLI while MOSS retains stage ordering and verdicts. Candidates are verified by replaying the batch against the candidate image in ephemeral trial workers, then promoted via user-consent-gated, in-place container swap with health-probe-gated rollback. On OpenClaw, MOSS lifts a four-task mean grader score from 0.25 to 0.61 in a single cycle without human intervention.

- **核心问题**：现有的自主智能体系统在部署后处于静态，无法从用户交互中学习，重复性故障需等待人工更新修复。虽然近年来出现了自我进化的智能体，但所有进化仅限于文本可变的工件（如技能文件、提示配置、记忆模式、工作流图），而智能体的核心框架（路由、钩子排序、状态管理、派遣等）仍保留在代码层面，无法通过文本修改触及。
- **整体含义**：文本层修改只能改变智能体“思考”的内容，无法改变框架“决定”的行为。当故障源自框架层（如消息路由错误、钩子执行顺序问题、会话状态损坏）时，任何文本工件修改都无法修复。作者论证源码级适应（Source-Level Adaptation）是更通用的媒介：它是图灵完备的，是文本可变范围的严格超集，能确定性执行，且不会因长上下文漂移而退化。

## 2. 方法论：核心思想、关键技术细节与流程

### 核心思想
- MOSS 实现源码级自我重写，将进化范围从文本工件扩展到智能体框架，通过自动收集生产故障证据、执行确定性多阶段流水线、委托外部编码智能体修改代码、在临时容器中回放验证候选版本，最后通过用户同意门控的就地容器替换部署生产环境。

### 关键技术细节
- **证据收集**：通过两种渠道获取生产故障证据：
  - 后台定期扫描（`moss evo catch-up`）自动分割会话日志并标记弱/缺失关键点。
  - 用户可在对话中直接标记不满意的轮次（`moss evo flag`）。
  - 每个对话维护一个开放批次，达到阈值（默认8块）后封闭并触发进化。
- **进化循环**：基于批次的定向进化，非随机探索。先运行基线评估锁定关键点集，然后迭代执行七阶段流水线，直至批次被修复或达到上限。
- **七阶段流水线（每迭代）**：
  1. **定位 (Locate)**：读取基线轨迹和批次故障，生成诊断（不提出修复）。
  2. **计划 (Plan)**：确定根本原因和修复方案（修改哪些文件、添加何逻辑）。
  3. **计划审查 (Plan-Review)**：质量门控，批准/拒绝计划，可与计划阶段交替进行。
  4. **实现 (Implement)**：在代码库中作为单个 git 提交写入代码。
  5. **代码审查 (Code-Review)**：质量门控，审查 diff 是否匹配计划。
  6. **任务评估 (Task-Evaluate)**：在构建和试运行后，对每个任务按四个等级（强/充分/弱/缺失）评分。
  7. **裁决 (Verdict)**：生成四种决策之一：已收敛、需要更多工作、模型天花板、架构天花板。
- **代码修改**：MOSS 通过确定性状态机管理进化调度和决策，具体编辑委托给可插拔的外部编码智能体 CLI（支持 Claude Code、OpenAI Codex、DeepSeek-TUI、OpenCode）。
- **验证机制**：构建候选镜像后，启动 N 个短生命周期试运行容器（与生产环境隔离），自动回放批次任务，暴露运行时故障。
- **部署与回滚**：收敛后用户需通过对话触发 `moss evo apply`，主机守护进程执行就地容器替换。使用 90 秒健康探测窗口（每 5 秒检查心跳、容器运行状态、两个 CLI 探测），连续三次通过则提交，否则回滚至上一次良好镜像。用户状态卷在替换中保持挂载，确保数据不丢失。

## 6. 主要结论与发现

- MOSS 在单次迭代中，无需人工干预将四任务平均 claweval 评分从 0.2526 提升至 0.6100（增幅 +0.3574）。其中 T138（补货链检查英文版）取得最大提升，从 0.2090 升至 0.9049（三次试验均超过 0.75 通过阈值）。SLA 合规审计任务从 0.25-0.33 区间提升至约 0.53-0.55。

- 验证了源码级自我进化可以修复框架级故障，这是先前文本层进化方法无法触及的。

## 7. 优点

- **进化范围更广**：MOSS 是首个覆盖智能体框架（路由、钩子、状态）的自进化系统，是文本层范围的严格超集。
- **确定性执行**：代码修改生效不依赖于基模型是否服从提示，效果不随上下文漂移而退化。
- **生产友好**：进化锚定真实用户故障证据而非人工基准；验证使用隔离容器回放；部署有用户同意门控和健康探测回滚，降低生产风险。
- **模块化与可插拔**：编码智能体通过四方法接口抽象，可替换供应商；进化阶段可配置深度（轻/标准/深度）。
